migration plan · 2026-05

Rewriting the pastedev frontend in Dioxus, without sacrificing a feature.

The current SPA is ~3,200 lines of Vue 3 + Pinia + vue-router, bundled with Vite and embedded by the Rust server via rust-embed. This plan replaces it with a Dioxus 0.7 web app compiled to WASM, keeping the entire feature surface — three snippet types, the inline-highlight editor, sandboxed HTML preview, auth, admin approval, API keys, setup wizard — and the same single-binary deployment story.

What this gets us

  • One language for the whole stack. Server, CLI, MCP, and SPA in Rust. Shared types from pastedev-core finally cross the wire on both sides.
  • End-to-end type safety. The API client mirrors handler signatures in the server crate — drift is a compile error, not a runtime HttpError.
  • Cheaper future renderer targets — for parts of the app. The Shell, auth pages, dashboard, admin queue and API-keys page are vanilla Dioxus components and would port to desktop / mobile / native renderers without changes. The editor (textarea + transparent-overlay + JS worker) and the HTML view (sandboxed iframe + cross-origin postMessage) are DOM-bound; those would need redesign before any non-web target. So: not "free for the whole app", but materially less work for ~70% of it.
  • Smaller toolchain. Drop node, pnpm, vue-tsc, Vite from the build. dx + cargo cover everything.

What it costs

  • WASM ships more bytes than DOM frameworks, and the gap is real. Upstream cites ≈70 kB gz for the bare Dioxus runtime, but that's the framework alone. With reqwest, web-sys, serde, pulldown-cmark, tracing-wasm and our application code, a realistic initial fetch lands at ≈300–600 kB gz vs. the current ~120 kB Vue bundle — bigger by 2–5×. Wasm-Split (new in 0.7) lazy-loads per-route chunks so non-editor routes ship less; we benchmark after phase 1 to calibrate. See R1.
  • Syntax highlighting stays in JS. highlight.js's 200-language auto-detect has no Rust equivalent we can ship in WASM. We keep the worker as-is and call it from Rust through wasm-bindgen — see editor.
  • One-off engineering cost. Estimated 5–7 working days for full parity, then 1–2 days of polish. See rollout.
