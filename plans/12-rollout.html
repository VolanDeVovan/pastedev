Rollout plan

Four phases, all on the rust_frontend branch. The Vue tree stays in place until phase 4 cuts over. Each phase ends with a runnable, testable state.

Phase map

phase 1
Foundation
crates/web skeleton compiles. Server still embeds the Vue dist.
phase 2
Read-only paths
All three view pages work end-to-end; non-approved flows + status page done.
phase 3
Author + admin
Editor, dashboard, admin queue, API keys. Full parity with Vue.
phase 4
Cutover
Flip the embed folder, delete web/, PR back to main.
timing is calibrated for "first Rust-WASM SPA in this project"
Anyone who has shipped a Dioxus 0.7 app of comparable surface in the past can probably hit the lower bound. Anyone treating this as their first non-trivial WASM project should expect the upper bound, especially for the editor (R8 in chapter 13). The day-by-day calendar at the bottom is the optimistic shape; the day-range per phase below is the realistic one.

Phase 1 — foundation (1.5–3 days)

Goal: "cargo build --workspace succeeds and renders 'hello' in a browser."

  1. Add crates/web to the workspace, with the dependency list from chapter 02.
  2. Stand up Dioxus.toml, tailwind.config.js, styles/input.css (copy from web/src/style.css).
  3. Wire mprocs: add tailwind + web procs, keep the old web proc temporarily so both work side-by-side.
  4. Land the app.rs + route.rs skeleton with the Route enum and a placeholder per variant.
  5. Land config.rs and verify the runtime config block parses.
  6. Land state/auth.rs, state/toast.rs, the api client.rs + error.rs + auth.rs.
  7. Smoke test: build, navigate to localhost:5173, see "hello from dioxus" rendered inside the server's index.html shell (server picks up crates/web/dist/ via a new build flag, see below).

During phase 1, the server still embeds web/dist/ by default. We add a cargo feature flag on pastedev-server — dioxus-spa — that swaps the embed folder when enabled: 

// crates/server/src/assets.rs
#[derive(RustEmbed)]
#[cfg_attr(not(feature = "dioxus-spa"), folder = "../../web/dist/")]
#[cfg_attr(feature = "dioxus-spa",     folder = "../../crates/web/dist/")]
#[exclude = ".gitkeep"]
pub struct Assets;

Dev runs cargo run -p pastedev-server --features dioxus-spa; the existing Vue dev path keeps working without the flag. Phase 4 removes the flag and deletes the Vue branch.

Phase 2 — read-only paths (2–4 days)

Goal: "a freshly registered user can browse snippets they didn't create."

  1. Land components/shell.rs + guarded.rs + form_field.rs.
  2. Land pages/signin.rs, register.rs, pending.rs, rejected.rs, setup.rs, status.rs.
  3. Land views/code.rs, markdown.rs, html.rs + the shared use_snippet helper.
  4. Land editor/highlight.rs + bundle the JS worker as an asset.
  5. Land lib/markdown.rs.
  6. Smoke-test paths: /setup, /status, /signin, /register, /pending, /c/:slug, /m/:slug, /h/:slug.

At the end of phase 2, an approved user can sign in, see a stub /dashboard placeholder, and click through to view their existing Vue-created snippets. The editor is still placeholder.

Phase 3 — author + admin (3–6 days)

Goal: "feature parity. No need to ever load the Vue SPA again."

  1. Land editor/mod.rs as three landable slices, in order:
    1. MVP-1 · textarea only. Toolbar + plain textarea + cmd-enter publish + edit-mode prefill. No overlay, no worker. Already publishable; visually a regression (no syntax color), so it lives behind the dioxus-spa feature flag.
    2. MVP-2 · overlay with worker. Hook up the highlight worker; paint hljs output into the overlay on every worker reply. Cold-start prepaint via escape_html(body) so the first frame is never blank. No cache yet — every keystroke flashes once for the round-trip.
    3. MVP-3 · cache + fast paths. Append / backspace / mid-edit / kind-flip paths from chapter 06. The "no flash on keystroke" parity bullet only ticks here.
    Each slice is reviewable independently. If the calendar tightens, MVP-2 is the minimum bar for cutover; MVP-3 can land as a follow-up PR after the Vue tree is deleted — the only regression is a 1-frame color flash, no functional loss.
  2. Land pages/dashboard.rs with the filter tabs, search, pagination.
  3. Land pages/admin.rs with the queue, the six actions, and the approve/reject modals.
  4. Land pages/api_keys.rs with create/list/revoke + the install/auth/publish/MCP panes.
  5. Land components/modal.rs + toast_dock.rs + the matching CSS transitions.
  6. Walk through every box in chapter 14; file follow-ups for any drift.

Once the checklist is fully ticked, we're ready for cutover. The Vue tree at web/ still lives — phase 4 deletes it.

Phase 4 — cutover (0.5–1 day)

  1. Remove the dioxus-spa feature gate on the server; only crates/web/dist/ is embedded.
  2. Delete web/ entirely.
  3. Remove the old build-web-dev / vue node deps from justfile and the Vue stage from the Dockerfile.
  4. Update the mprocs config to drop the old web proc.
  5. Run cargo test --workspace and the wasm-bindgen e2e smoke test.
  6. Manual QA: re-run the parity checklist on a clean Docker image.
  7. Open the PR to merge rust_frontend into main.

The editor is the schedule risk — overlay + cache + worker + cmd+enter + scroll-sync against textarea is genuine complexity. The MVP staging above lets us cut MVP-3 from the critical path if days 4–5 spill; MVP-2 is the cutover minimum.

Branching strategy

Work lives on rust_frontend (already created). Each phase lands as a series of normal commits on the branch — no sub-branches. Phase 3 may benefit from one rebase-against-main mid-flight to pick up any server-side fixes.

No long-lived hybrid state. The feature gate exists only inside the branch during phases 1–3 so we can flip back to Vue without a revert if a smoke test fails mid-way. Phase 4 removes the gate; the PR to main is a single, large but reviewable change.

Backout plan

Before the phase-4 PR: do nothing — the feature flag defaults to off. After the phase-4 PR: git revert the merge commit. The Vue tree is still in git history, the rust-embed path is one line. No data migrations to undo; no API surface changes to back out.

Communication / review

  • One PR per phase if reviewers prefer it; one fat PR otherwise. Either way the diff is mostly additive in phases 1–3.
  • The phase-4 PR carries the deletion of web/ — call it out in the PR description so reviewers don't have to scroll past 3k lines of deleted Vue to find the change.
  • The parity checklist (chapter 14) doubles as the PR's acceptance section.
