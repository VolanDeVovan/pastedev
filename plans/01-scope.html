chapter 01

Scope & non-goals

The migration replaces exactly one thing: the Vue SPA that lives under web/. Server, database, CLI, MCP — none of them move.

In scope

areatodayafter migration
SPA shell & routing Vue 3, vue-router 4 Dioxus 0.7, dioxus-router
global state Pinia (auth, toast) Signals + a global AuthContext
api client handwritten fetch in web/src/api/ reqwest-WASM in crates/web/src/api/
editor EditorView.vue + cached-prefix overlay Dioxus component with the same overlay strategy
view pages ViewCode/ViewMarkdown/ViewHTML.vue Dioxus components (one file each)
auth pages Sign in, register, pending, rejected, status, setup Same set, same flows, same routes
dashboard & admin DashboardView, AdminView, ApiKeysView Identical UX, same list/filter behaviour
styling Tailwind v4 + @theme tokens Same — keep Tailwind, just point it at .rs sources
runtime config injection <script id="pastedev-config"> in index.html Unchanged — Dioxus reads the same DOM node on boot
syntax highlight highlight.js in a Web Worker Same worker, called from Rust via wasm-bindgen
markdown rendering markdown-it + hljs fences pulldown-cmark (already in pastedev-server) in WASM
iframe sandbox + size reporter Server-injected reporter, client postMessage Unchanged on the server; client postMessage moves to wasm-bindgen

Out of scope

areawhy we don't touch it
server crate All API routes, auth extractors, the setup gate, security headers stay byte-for-byte the same. Only the rust-embed folder argument changes — see 11.
database schema & migrations The wire format is the contract. The DB doesn't know we changed renderers.
pastedev-cli & MCP Different binary. Unaffected.
pastedev-core types The wire-format crate is consumed by server today and by the new WASM client tomorrow. We start using it more, not less — but no schema changes.
deploy story Still one pastedev-server binary with the SPA embedded. The Dockerfile gains a dx build stage instead of pnpm run build — see 11.
design / visual identity The design tokens (--color-bg, --color-accent, etc.) and the screen layouts from plan/design/project/screens.jsx stay. We're not redesigning, just re-implementing.
SSR / hydration Dioxus fullstack with server functions is tempting, but the existing axum server already owns routing, auth, rate limiting, etc. Reading cookies/sessions during Dioxus server-render would duplicate every middleware. CSR keeps the layering clean — see 02.

Non-goals worth calling out

  • No new features. If the box isn't on the parity checklist, we don't add it during the migration. Feature work and migration work cleanly separated keep the diff reviewable.
  • No "pure-Rust syntax highlighter". We considered syntect, tree-sitter-highlight, and pure-WASM equivalents. None match hljs's auto-detect-across-200-languages behaviour, which is the whole point — see 06.
  • No removal of Tailwind. Stylo / global CSS scoped to components was considered. Tailwind v4 already works file-agnostically (it scans content: globs); pointing it at **/*.rs just works — see 10.
  • No new dependencies on the server. All new crates are in crates/web (or shared via pastedev-core). Server Cargo.toml changes are limited to the embed folder path.

Functional surface — the contract we keep

These are the user-visible features the new SPA must reproduce. They're the acceptance criteria — anything missing here is a parity bug. The full enumerated checklist lives in chapter 14; this list groups them by surface so we can sanity-check coverage.

  1. Visitor flows. Browse /c/:slug, /m/:slug, /h/:slug without an account; setup gate redirects to /setup on fresh instances; /status renders on a borked instance.
  2. Account flows. Register → pending → approved (or rejected). Sign in/out. Cookies are credentials: include; the SPA never sees the token.
  3. Authoring. Type-tab (code/md/html), filename, body up to 1 MB. Cmd/Ctrl+Enter publishes. Edit mode for own snippets. Delete confirmation modal.
  4. Inline highlight. Code & HTML editor show colored syntax under the textarea, no flash on keystroke, debounced worker.
  5. Markdown preview. Split pane on desktop, stacked on mobile, 120ms debounce.
  6. Snippet views. Headers with copy-link / copy-raw / open / edit / delete actions, line-gutter on code, sandboxed iframe with auto-sizing on HTML, markdown article.
  7. Admin queue. Approve / reject / suspend / restore / promote / demote with confirmation modals on destructive ones.
  8. API keys. Create / revoke with the one-shot reveal of pds_live_*, plus the install / authenticate / publish / MCP example pane.
  9. Setup wizard. Env-check polling + first-admin form, auto-redirect to / when complete.
  10. Mobile shell. Bottom tab bar appears for approved users, padding-bottom on main accounts for it.
  11. Toast dock. Bottom-right stack, error toasts last 6s, others 4s.
  12. Status page. Health-check rows, polling cadence 5s when ok / 15s when not.
acceptance bar
The PR that flips the embed folder from web/dist/ to the new crates/web/dist/ ships only when every box in chapter 14 is ticked and the Vue tree is deleted in the same commit. No long-lived hybrid.
