chapter 02

Dioxus stack picks

Dioxus has multiple shipping modes and several optional crates. This chapter pins exactly which pieces we use, and why.

Rendering mode: client-side WASM (not fullstack)

Dioxus 0.7 supports three modes: web (pure CSR, WASM in the browser), fullstack (server functions + hydration, runs on axum), and desktop/mobile/native. We pick web.

Router: built-in via the router feature

In 0.7 the router ships as a feature of the main dioxus crate — no separate dioxus-router dependency to track. Enable it with features = ["web", "router"] and import from dioxus::prelude::*. The router is typesafe: you describe routes as a #[derive(Routable, Clone)] enum, and the compiler refuses to navigate to a path that doesn't exist. It supports nested routes (#[nest]), shared layouts (#[layout]), dynamic segments (:id), query segments (?:name&:other) — everything vue-router gives us, plus compile-time checking.

State management: signals + a global context (+ Stores for nested cases)

Dioxus 0.7's Signal<T> + use_context cover what Pinia gave us. We don't need a separate state library.

pinia (vue)

const auth = useAuthStore();
auth.user        // ref<UserPublic|null>
auth.isAdmin     // computed
await auth.boot();
await auth.login({...});

dioxus signals + context

let auth = use_auth();
auth.user.read()           // Option<UserPublic>
auth.is_admin()             // memo
auth.boot().await;
auth.login(...).await;

The pattern: a single AuthState struct of signals is provided once at the root with use_context_provider; every component grabs it with a thin use_auth() helper that calls use_context. Toasts work the same way — a ToastQueue context with a single Signal<Vec<Toast>>.

For async-derived state (the auth boot, the admin user list, the snippet fetch on view pages) we use use_resource, Dioxus's SWR-style primitive. It tracks the closure's reactive dependencies and re-runs the future when they change — clean fit for "re-fetch when the slug param changes".

Stores are a 0.7 addition for nested reactive state (think Pinia stores but with structural reactivity — writing store.items.push(...) only notifies readers of items). We don't need them for this rewrite — our state graph is shallow — but they're a good escape hatch if the admin list ever grows into something that needs per-row reactivity.

HTTP: reqwest with the wasm32-unknown-unknown backend

reqwest builds for WASM, sends through fetch under the hood, supports JSON, and accepts the per-request RequestBuilder::fetch_credentials_include() we need so the session cookie travels with same-origin and cross-origin calls. It's the right level of abstraction — one error type, one client, one builder. The Dioxus 0.7 official data-fetching tutorial uses reqwest, so we follow the well-paved road. Detailed wrapper in chapter 05.

cookies + CORS in WASM
Browser fetch only sends cookies on cross-origin requests if both the client passes credentials: include AND the server's CORS headers explicitly allow it. The existing server already does this for CORS_ALLOWED_ORIGINS — we just need to keep that list in sync with the dev origin dx serves from (see 11).

Alternatives considered and dropped:

  • gloo-net — thinner, ~30 kB smaller, but doesn't share an error type with what the server uses (reqwest::Error), and we'd reimplement everything reqwest already does (json, query-string, redirects).
  • Raw web_sys::Request — works, but every call site becomes 40 lines of boilerplate.

Markdown: pulldown-cmark

The server already depends on it, builds clean to wasm32, and the output html is functionally equivalent to markdown-it's for our content (headings, lists, blockquotes, fenced code, tables, GFM autolinks if we turn on the feature). We do not let it render raw HTML — pulldown-cmark default is to escape, which is what we want.

Syntax highlight: keep highlight.js, call it from Rust

No pure-Rust highlighter we evaluated covers hljs's auto-detect across ~200 languages. syntect uses TextMate grammars and needs an explicit language. tree-sitter-highlight requires per-language WASM grammars. Both would make the bundle larger than just shipping the existing JS worker.

Strategy: keep web/src/workers/highlight.worker.ts as-is (or move to crates/web/js/highlight.worker.js), let dx copy it into the bundle as a static asset, and call it from Rust through a tiny wasm-bindgen JS glue file. Detailed in chapter 06.

Build tool: dx (Dioxus CLI)

We use dx serve in dev (with element & asset hot-reload by default; pass --hotpatch for the new Subsecond Rust-code hot-patching, currently flagged experimental) and dx build --release for production. The output goes to crates/web/dist/ and is embedded by the server crate via rust-embed.

Tailwind: keep it

Tailwind v4 is content-agnostic — it scans whatever globs you tell it to. We just change content from **/*.{vue,ts} to **/*.rs and the existing classnames keep working. Full story in chapter 10.

What we deliberately don't add

  • No Fermi / Redux-style global store. Signals + context are enough for two small stores.
  • No Dioxus SSR / Liveview. CSR is the decision; mixing in a partial-SSR path doubles the surface for no payoff.
  • No custom highlighter. See above — hljs through interop wins.
  • No leptos, yew, or sycamore. The framework choice is settled — see the rust_frontend branch memory.
