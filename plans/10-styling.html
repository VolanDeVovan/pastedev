chapter 10

Styling story

Tailwind v4 stays. The @theme tokens move with it. Classnames are already strings — they don't care that the host framework changed. Only one knob turns: the content scan glob.

Decision: keep Tailwind v4 + let dx drive it

dioxus 0.7 ships first-class tailwind support
Per the upstream guide (dioxuslabs.com/learn/0.7/guides/utilities/tailwind/), dx auto-detects a tailwind.css at the project root and automatically downloads and runs the Tailwind CLI on every dx serve / dx build. We lean into that instead of running a parallel npx tailwindcss --watch process. Net effect: no node, no pnpm, no separate mprocs proc — the entire styling pipeline rides on dx.

Alternatives considered:

optionprosconsverdict
Stylo / Dioxus inline styles One language, type-checked classes Have to rewrite every class. Dynamic-class composition (e.g. active states) gets noisier. no
Hand-rolled global CSS Smallest bundle Would discard 100+ Tailwind utility classes already tuned to the design tokens. no
Tailwind via external CLI / parallel watcher What we'd have written before reading the 0.7 guide Adds a node toolchain to dev and to CI for no reason; duplicates what dx already automates. no
Tailwind v4 driven by dx's built-in CLI integration Zero churn to classnames; @theme tokens carry over verbatim; one toolchain (no node/pnpm); HMR for free None worth listing yes

Project layout

crates/web/ ├── tailwind.css ← THE convention: dx auto-runs the Tailwind CLI on this file │ (entry — tokens, @source globs, layer/keyframe rules, .md-preview) ├── assets/ ← compiled output lands here (dx writes assets/tailwind.css) │ ├── favicon.svg │ ├── fonts/... │ └── highlight.worker.js ├── index.html ← (optional; can also link the stylesheet via document::Stylesheet in rsx) ├── src/ ← Dioxus components (scanned by Tailwind via @source) ├── Dioxus.toml └── Cargo.toml

The single tailwind.css at the crate root

In Tailwind v4 the config file (the old tailwind.config.js) is replaced by CSS directives inside the entry stylesheet. So we keep one file, and that file is our config: 

/* crates/web/tailwind.css — the SINGLE source of truth that dx watches */
@import "tailwindcss";
@import "highlight.js/styles/atom-one-dark.css";

/* Replaces content[] in old tailwind.config.js. */
@source "./src/**/*.{rs,html}";
@source "./index.html";

/* Same design tokens we ship today in web/src/style.css. */
@theme {
  --color-bg:            #0d1620;
  --color-bg-deep:       #0a1119;
  --color-border:        #1a2433;
  --color-border-strong: #2a3548;
  --color-text:          #d8dfeb;
  --color-text-dim:      #8b97ad;
  --color-text-muted:    #6a7a92;
  --color-text-faint:    #3d4a60;
  --color-accent:        oklch(0.82 0.11 165);
  --color-warn:          oklch(0.82 0.13 75);
  --color-danger:        oklch(0.75 0.16 25);
  --color-danger-border: oklch(0.45 0.12 25);
  --font-mono: 'JetBrains Mono', ui-monospace, SFMono-Regular, Menlo, monospace;
  --font-sans: 'Inter', ui-sans-serif, system-ui, sans-serif;
}

@layer base { /* body font, scrollbar hide, .hljs reset — same as today */ }

.md-preview { /* unchanged article ruleset */ }

@keyframes paste-pulse {
  0%, 100% { opacity: 0.3; transform: scale(0.9); }
  50%      { opacity: 1;   transform: scale(1); }
}
why globbing .rs works
Tailwind v4's @source scanner is regex-based — it greps for class strings inside any text. RSX class attributes look like class: "..." in source; the literal strings tailwind finds are the exact same set that's in .vue files today. No Dioxus-specific plugin needed.

Linking the compiled stylesheet

dx writes the compiled stylesheet to assets/tailwind.css. The idiomatic 0.7 way to pull it into the page is the document::Stylesheet component with the asset!() macro inside app.rs — same as the upstream guide: 

// crates/web/src/app.rs
use dioxus::prelude::*;

const TAILWIND_CSS: Asset = asset!("/assets/tailwind.css");

#[component]
pub fn App() -> Element {
    rsx! {
        document::Stylesheet { href: TAILWIND_CSS }
        // providers + <Router> + <ToastDock />
        ...
    }
}

Routing-as-asset (the asset!() macro) hashes the filename automatically — same cache-busting story we had with Vite, just one rung lower in the toolchain.

The hljs theme

The current SPA imports highlight.js/styles/atom-one-dark.css from node_modules. Once we drop pnpm from crates/web/ there's no node_modules anymore — so we vendor it: drop the file as crates/web/styles/atom-one-dark.css and adjust the @import to a relative path. ≈4 kB of static CSS, one less dependency in dev.

Build & dev: same single command

Because dx owns the tailwind step, both dev and CI become one command: 

# dev (under mprocs)
cd crates/web && dx serve

# release build
cd crates/web && dx build --release --platform web

mprocs only needs the web and server procs now — no separate tailwind proc: 

# mprocs.yaml — clean two-proc setup
procs:
  web:
    shell: "cd crates/web && dx serve --port 5173"
  server:
    shell: |
      DATABASE_URL=...
      cargo run -p pastedev-server
  db-logs:
    shell: "docker compose logs -f db"

RSX class composition — the conditional patterns

Vue's :class="[ 'a', cond ? 'b' : 'c' ]" turns into one of these in Dioxus:

vue

:class="[
  'pb-0.5 border-b',
  isActive('home')
    ? 'text-text border-accent'
    : 'text-text-muted'
]"

dioxus rsx

class: if route.is_home() {
    "pb-0.5 border-b text-text border-accent"
} else {
    "pb-0.5 border-b text-text-muted"
}

For class soup we use classnames! — a small helper macro in lib/classnames.rs: 

macro_rules! classnames {
    ($($cls:expr),+ $(,)?) => {{
        let mut s = String::new();
        $(
            let c: &str = $cls;
            if !c.is_empty() { if !s.is_empty() { s.push(' '); } s.push_str(c); }
        )+
        s
    }};
}

// usage
class: classnames!(
    "pb-0.5 border-b",
    if active { "text-text border-accent" } else { "" },
    if !active { "text-text-muted border-transparent hover:text-text" } else { "" },
)

Responsive — Tailwind's md: prefix stays

Every md:flex-row / md:px-7 / md:hidden we have today keeps working because Tailwind v4 generates the same utilities regardless of who wrote the consumer. Mobile-first stays mobile-first.

Where we deliberately differ from Vue

  • No more v-html for the markdown article — we use Dioxus's dangerous_inner_html. Identical semantics; just named honestly.
  • No more component-scoped <style scoped> blocks (the Modal had one). We move those handful of rules into input.css as a regular global rule with a unique class prefix.
  • The md-preview article styles stay global — they're applied to user-rendered HTML and we want them everywhere.

Color tokens — unchanged

The exact same --color-* palette feeds Tailwind v4 via @theme. Token names (and therefore class names like bg-bg-deep, text-text-muted, border-border-strong) carry over verbatim. The doc site you're reading uses the same tokens, on purpose.
