chapter 11

Server embed & build

One line in the server crate flips. The justfile, Dockerfile, and mprocs config gain a dx step and lose the pnpm vue step. The runtime-config injection contract stays exactly the same.

The single-line server change

// crates/server/src/assets.rs
#[derive(RustEmbed)]
// before
#[folder = "../../web/dist/"]
// after
#[folder = "../../crates/web/dist/"]
#[exclude = ".gitkeep"]
pub struct Assets;

That's it for production. The rest of the server crate is unchanged: http/shell.rs still finds </head> and splices the runtime config; http/mod.rs still slaps the same security headers on responses; http/snippet_meta.rs still injects per-snippet OG/Twitter tags above the runtime config script.

The runtime config contract — invariant

The server emits exactly one element with a known id and a known JSON shape: 

<script id="pastedev-config" type="application/json">
  {"apiBaseUrl":"", "publicBaseUrl":"https://paste.example.com", "appName":"pastedev"}
</script>

The Dioxus client reads it on boot: 

// crates/web/src/config.rs
use serde::Deserialize;
use wasm_bindgen::JsCast;

#[derive(Deserialize, Clone, Debug)]
pub struct PasteConfig {
    #[serde(rename = "apiBaseUrl")]    pub api_base_url:    String,
    #[serde(rename = "publicBaseUrl")] pub public_base_url: String,
    #[serde(rename = "appName")]        pub app_name:        String,
}

pub fn config() -> &'static PasteConfig {
    static C: OnceLock<PasteConfig> = OnceLock::new();
    C.get_or_init(|| {
        let doc = web_sys::window().and_then(|w| w.document());
        let txt = doc
            .and_then(|d| d.get_element_by_id("pastedev-config"))
            .and_then(|el| el.text_content())
            .unwrap_or_default();
        serde_json::from_str(&txt).unwrap_or_else(|_| PasteConfig::defaults())
    })
}

Title-tag contract — also invariant

http/shell.rs has a constant STATIC_TITLE = "<title>pastedev</title>" that it strips when injecting a per-snippet title. Our new index.html ships with that exact title element so the snippet-meta injection path works unchanged. There's even a unit test pinning the constant — we don't touch that test.

Dev: mprocs, two ports

Same model as today — Rust server on :8080, web dev server on :5173, browser hits :5173, axum gets reverse-proxied for /api and /{c,m,h}/:slug/raw. dx serve has its own proxy config; alternatively we keep using vite-style proxying via dx's built-in dev-server proxy file: 

# crates/web/Dioxus.toml
[application]
name = "pastedev-web"

[web.app]
title = "pastedev"

[web.watcher]
reload_html = true
watch_path = ["src", "assets", "styles"]

[web.proxy]
backend = "http://localhost:8080"
endpoints = [
    "/api",
    "/c/{slug}/raw",
    "/m/{slug}/raw",
    "/h/{slug}/raw",
]

[web.resource.dev]
style = ["/assets/main.css"]   # tailwind output goes here in dev

[web.wasm_opt]
level = "z"
CORS allow-list — keep in sync
In dev, axum has CORS_ALLOWED_ORIGINS=http://localhost:5173 so that cross-origin auth works. If dx ends up serving on a different port (it defaults to :8080), update the mprocs env entry. Same-origin in prod (the server serves the SPA itself) needs no CORS.

Justfile changes

Since Dioxus 0.7 runs Tailwind for us, dx build is the only command we need:

before

build-web:
    cd web && pnpm install --frozen-lockfile \
        && pnpm run build

dev: build-web-dev db-up _ensure-mprocs
    .tools/bin/mprocs --config mprocs.yaml

build-web-dev:
    cd web && [ -d node_modules ] \
        || pnpm install

after

build-web:
    cd crates/web && dx build \
        --release --platform web
    # dx auto-runs tailwindcss on
    # crates/web/tailwind.css —
    # no node/pnpm step needed.

dev: db-up _ensure-mprocs _ensure-dx
    .tools/bin/mprocs --config mprocs.yaml

_ensure-dx:
    @[ -x .tools/bin/dx ] || \
        cargo install \
            --root .tools \
            --locked dioxus-cli

Dockerfile changes

# Dockerfile — three stages. Stage 1 only needs the Rust toolchain
# (+ dx). No node, no pnpm — dx fetches and drives the tailwind CLI on
# its own when it sees crates/web/tailwind.css.

# stage 1 — build the SPA (dx runs tailwind internally)
FROM rust:1.90-alpine AS web
RUN apk add --no-cache musl-dev openssl-dev pkgconfig && \
    cargo install --locked dioxus-cli && \
    rustup target add wasm32-unknown-unknown
WORKDIR /src
COPY Cargo.toml Cargo.lock* ./
COPY crates/ ./crates/
WORKDIR /src/crates/web
RUN dx build --release --platform web

# stage 2 — build the Rust server, embedding crates/web/dist
FROM rust:1.90-alpine AS rust
RUN apk add --no-cache musl-dev openssl-dev pkgconfig
WORKDIR /src
COPY --from=web /src ./
COPY .sqlx/ ./.sqlx/
ENV SQLX_OFFLINE=true
RUN cargo build --release -p pastedev-server

# stage 3 — runtime (unchanged)
FROM alpine:3.20
RUN apk add --no-cache ca-certificates
COPY --from=rust /src/target/release/pastedev-server /usr/local/bin/pastedev-server
USER nobody
EXPOSE 8080
ENTRYPOINT ["/usr/local/bin/pastedev-server"]

Notable: stage 1 drops nodejs / npm / pnpm entirely. dx downloads the Tailwind CLI on first invocation and caches it. Stages 1 and 2 share a base image so the Rust toolchain is reused. We still run with SQLX_OFFLINE=true — the .sqlx/ cache is unaffected by the SPA swap.

Cargo.dist note

The release pipeline (cargo-dist in Cargo.toml) is wired to pastedev-cli only. pastedev-server has dist = false implicitly (it has the rust-embed macro that needs web/dist/ at compile time). After we switch the embed folder to crates/web/dist/, the same flag still keeps it out of cargo-dist's path. No release-pipeline edits needed.

The flake.nix packages

The Nix flake builds both binaries. pastedev-cli stays unchanged. pastedev-server needs dioxus-cli and the wasm32-unknown-unknown Rust target in its build inputs — and nothing else. Since dx auto-runs the Tailwind CLI (see chapter 10), no nodejs / pnpm in the build inputs. The Tailwind binary dx downloads is a standalone Go binary, not a node module.

  • dioxus-cli (from nixpkgs if available; otherwise a cargo-installed derivation pinned to the version in flake.lock)
  • rustup with the wasm32-unknown-unknown target, or the equivalent Crane / fenix wiring
  • Internet access in the build sandbox is the gotcha — dx fetches the Tailwind CLI on first run. If the Nix build sandbox blocks the network, we either (a) preseed the Tailwind binary in $XDG_CACHE_HOME/dioxus as a fixed-output derivation, or (b) accept that the flake needs --impure for the SPA build step.

Net effect on existing Nix users: nix build .#pastedev-server gets slower (one extra compile step) but produces the same artifact. The existing feedback_nix_for_missing_tools memory still applies — Nix is the fallback for missing tools.

What stays byte-for-byte the same on the server

  • Every /api/v1/* route, request shape, response shape.
  • The STATIC_TITLE constant and the smoke test guarding it.
  • The runtime config JSON shape (apiBaseUrl, publicBaseUrl, appName).
  • The setup gate, the auth extractors, the CSP, the iframe sandbox CSP.
  • The HTML size-reporter snippet (HTML_SIZE_REPORTER in snippets/handlers.rs).
  • The fallback shell HTML for when the SPA isn't built yet.

What we'll need to add to pastedev-core

Today's pastedev-core exposes only what the CLI needs. To stop duplicating types into the new WASM client we extend it with the types listed in chapter 03. All additive; the server crate gains no edits — it already uses these types via its own internal modules.
