chapter 03

Crate & module layout

The new SPA lives in a fourth workspace crate, crates/web, sibling to core, server, and cli. File names mirror the old web/src/ tree so the diff is readable.

Tailwind v4 has no separate config file — its directives live inside the entry stylesheet. dx auto-detects crates/web/tailwind.css and runs the Tailwind CLI on every dx serve / dx build. The compiled output lands at crates/web/assets/tailwind.css, hashed and embedded by the server. Full layout of the stylesheet (tokens, @source globs, @layer rules, .md-preview) is in chapter 10.

One source of truth for wire types

Today the TypeScript types in web/src/api/types.ts mirror the Rust types in crates/core/ by hand. After the migration we drop the second copy and have the WASM client actually depend on pastedev-core:

before

// web/src/api/types.ts
export type Role = 'user' | 'admin';
export interface UserPublic {
  id: string;
  username: string;
  role: Role;
  status: UserStatus;
  created_at: string;
}
// drifts silently when the
// Rust side adds a field

after

// crates/web/src/api/types.rs
pub use pastedev_core::{
    // already exposed today
    Role, UserPublic, UserStatus,
    SnippetType, Scope, ErrorEnvelope,
    Snippet, SnippetListItem,
    CreateSnippetRequest, PatchSnippetRequest,
    ListSnippetsResponse,
    // added by the migration (see below)
    LoginRequest, RegisterRequest, UserEnvelope,
    AdminUserView, AdminListQuery, AdminListResponse,
    UserMutationResponse, ResetPasswordRequest,
    SetupCheck, SetupStatus, SetupAdminRequest, SetupAdminResponse,
    CreateKeyRequest, KeyView, KeyMintedView, KeyList,
};
// schema change in core = compile
// error here AND in the server
core crate work — concrete checklist
Today pastedev-core exposes only what the CLI/MCP needs. Everything else the SPA consumes currently lives in handler-local structs inside the server crate. The migration moves them into core (or adds them, if they don't exist on either side yet). The server keeps working because it now uses the same struct it used to declare locally.
typecurrently inmoves to
LoginRequest, RegisterRequest, UserEnvelope server/src/users/handlers.rs core/src/user.rs
AdminUserView, ListQuery, ListResponse, UserMutationResponse, ResetPasswordRequest server/src/users/admin.rs core/src/admin.rs (new module)
Check, SetupStatus, AdminRequest, AdminResponse server/src/setup.rs core/src/setup.rs (new module)
CreateRequest, KeyView, KeyMintedView, KeyList server/src/api_keys/handlers.rs core/src/api_key.rs (new module)

Strictly additive on the wire — same JSON shape on either side. But it's not free: each move is a small server-crate diff. Two things to double-check while moving:

  • Some server-local names collide with new module names (e.g. setup::AdminRequest vs. users::admin::ListQuery). Rename on the way in: SetupAdminRequest, AdminListQuery, AdminListResponse. The server's use sites are 1-line edits.
  • Confirm sqlx::query! macros don't infer their deserialize targets from these types (they shouldn't — these are response/request DTOs, not row types). If they do, just prepare after the move.

Budget half a day for the moves + a unit test in core that serializes each type to a JSON snapshot matching what the server emits today.

The index.html boot shell

The server's HTML rendering pipeline (http/shell.rs) finds </head> in index.html and splices the runtime config block before it. We keep exactly that contract. The new boot shell looks like: 

<!doctype html>
<html lang="en">
<head>
  <meta charset="UTF-8" />
  <link rel="icon" type="image/svg+xml" href="/favicon.svg" />
  <meta name="viewport" content="width=device-width, initial-scale=1.0" />
  <title>pastedev</title>
  <link rel="stylesheet" href="/assets/main.css" />
  <!-- server splices <script id="pastedev-config"> here -->
</head>
<body class="bg-bg text-text">
  <div id="main"></div>
  <script type="module" src="/assets/pastedev-web.js"></script>
</body>
</html>

dx writes the WASM glue, the JS shim, the wasm-bindgen output, and the hashed assets into crates/web/dist/. The server's STATIC_TITLE constant stays at <title>pastedev</title> — same as today — so the snippet meta injection path is unchanged.

Module split rationale

  • editor/ gets its own folder because the overlay + worker + cache logic is the gnarliest piece of the SPA. Keeping it isolated also lets us lazy-load it via Wasm-Split — see 13.
  • views/ are the public read-only pages (slug routes). pages/ are the account-area pages (signed-in / admin / setup). The Vue tree mixed both under views/; the split makes guard wiring easier — see 04.
  • state/ stays as a single folder (not split per signal) because there's only two stores. We'll resist the urge to add more "state files" — each component has its own local use_signals.
  • lib/ is the leaf-utility bin: stuff that takes inputs and returns outputs, no Dioxus dependencies.
  • js/interop.rs centralizes the wasm-bindgen-y bits so the rest of the codebase reads as idiomatic Rust.
