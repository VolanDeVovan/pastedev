R1 · WASM payload on mobile

The 70 kB number from upstream docs is the framework alone — cargo new + dioxus::launch(|| rsx! { "hi" }). Realistic budget after we add reqwest (≈40–60 kB), web-sys with the features in chapter 02 (≈30–80 kB depending on what gets monomorphized), serde + serde_json (≈30 kB), pulldown-cmark (≈40 kB), tracing-wasm (≈10 kB), plus application code: we should plan for 300–600 kB gz initial fetch. That's 2–5× the current Vue bundle (~120 kB gz). On 3G mobile this is a noticeable first-load tax, less so on 4G/Wi-Fi where the gap is sub-second.

Mitigations:

  • Use Wasm-Split on the editor route — it lazy-loads only when an approved user navigates to /. Same for admin (admins are <1% of sessions).
  • Tune [web.wasm_opt] level = "z" for release.
  • Lazy-load the highlight worker on first code or HTML view (markdown and view-only routes never need it).
  • Tripwire — calibrated, not aspirational: snapshot the gzipped initial fetch at end of phase 1 (skeleton + auth + Shell only) and at end of phase 2 (view pages). Phase-1 baseline tells us framework+plumbing cost; phase-2 baseline tells us per-route cost. Cutover acceptance: initial fetch on the un-authenticated landing path (e.g. /c/abc) ≤ 350 kB gz. The editor's authenticated payload can be larger if Wasm-Split is wired.
  • If the phase-1 baseline already exceeds 200 kB gz before any application code, that's the signal to revisit feature flags (the web-sys list is the usual culprit).

R2 · hljs worker interop

The cache and token-check logic is what makes the editor feel smooth. Misordered replies (a slow reply for the previous body arriving after a faster reply for the current one) would re-paint stale colors over the live text.

Mitigations:

  • The Rust side uses a monotonic request_id and discards any reply whose id is less than the current one — same scheme as useHighlight.ts.
  • One unit test per fast path (append / backspace / mid-edit / kind-flip), using a fake worker that just echoes the request id.
  • Manual: paste a 500 kB file, hold backspace; nothing should flicker.

R3 · Iframe size-reporter cycle

The Vue version solves this with SIZE_SLACK + SIZE_HYSTERESIS. Translating those constants verbatim keeps the feedback loop broken. The risk is forgetting them.

Mitigations:

  • Lift the two constants into the new module as named consts; comment links to the Vue source so the reasoning isn't lost.
  • Reproduce the test case in QA: an HTML snippet that itself observes its document size with a ResizeObserver and re-applies it.

R4 · reqwest cookies in WASM

reqwest's WASM backend delegates to fetch. Cookie behavior is governed by the browser. The fetch_credentials_include() method exists for this reason — we call it on every request.

Mitigations:

  • Phase 1's end-to-end smoke test exercises login → me → logout. If a cookie isn't getting attached, this fails.
  • If reqwest's WASM backend has a regression on a version bump, fall back to gloo-net for the call site (single point of change in client.rs).

R5 · Tailwind missing class strings

Tailwind v4's content scanner is regex-based; it greps for class-like tokens in source text. Strings produced at runtime via format!() aren't visible to it. Dynamic class composition has to be done with complete classnames as literals, not assembled values.

Mitigations:

  • Code review for any format!("text-{}", ...) patterns that try to build class strings.
  • If we need a dynamic palette (e.g. admin avatars), the literal strings live in a const array in .rs source — tailwind scans .rs as text, so it picks them up.
  • Tripwire: a safelist for the user-derived avatar classes if static inclusion isn't enough.

R6 · Router type-state quirks

Dioxus 0.7's Routable derive is strict about Clone + PartialEq on all variant fields. Optional query segments (?:next) map to Option<T>; catch-all segments map to Vec<String>.

Mitigations:

  • Land the route enum first; iterate compile errors before any UI is written.
  • Each variant gets a one-line round-trip test: assert_eq!(Route::ViewCode { slug: "abc".into() }.to_string(), "/c/abc");

R7 · Subsecond hot-patching stability

--hotpatch is opt-in and labeled experimental in 0.7. Element/asset hot-reload (no flag) is stable and covers most edits.

Mitigations:

  • Default to dx serve (stable). Use --hotpatch only when iterating on a single component.
  • Don't make CI depend on hot-patching.

R8 · Schedule slip on the editor

The editor is the schedule risk. Day 4 in the calendar is reserved for it; if it spills into day 5 we tighten dashboard + keys (both mostly composition).

Mitigations:

  • Land the editor in three stages: textarea-only (no overlay) → overlay with worker → overlay with cache. Each stage is publishable.
  • If stage 2 works but stage 3 (the cache) is fighting, ship without the cache for v1 — the only regression is a 1-frame flash on every keystroke, no functional loss. Backfill the cache after cutover.

R9 · Title-tag race with WASM mount

The server writes per-snippet <title> when the page is for a slug route. After the WASM client mounts, it could document.set_title(...) to something else.

Mitigation: we deliberately don't set the title from the client. The server-rendered title is correct for slug routes; for anything else the static pastedev is fine.

R10 · CSP friction

Server CSP for non-sandboxed paths is script-src 'self'. The new bundle ships under /assets/* — same origin — so the policy is satisfied for external scripts. The friction is inline: if dx emits an inline <script> for the WASM bootstrap, our CSP blocks it and nothing renders.

10-minute pre-flight before phase 1
Run a throwaway dx build --release --platform web on a stock dioxus 0.7 scaffold, open the generated dist/index.html, and grep for inline <script> tags. If any exist without a hash or nonce-able external form, we know we need either (a) a CSP nonce path in http/shell.rs or (b) a Dioxus.toml setting that externalizes the bootstrap. Cheaper to find this in 10 minutes than at end of phase 2 when nothing loads in prod.

Mitigations:

  • If dx emits an inline script for bootstrap, configure it to write to a file (preferred), or extend http/shell.rs to inject a per-response nonce + script-src 'nonce-xxx' 'self'.
  • The hljs worker is loaded with new Worker(WORKER_JS.path()) — that's a same-origin URL, satisfies script-src 'self'. No worker-src override needed.
  • The sandbox CSP on /h/*/raw is unrelated — it stays exactly as today.

R11 · Dev rebuild times

Vite HMR sits at < 200 ms for a typical edit; we relied on that for years. A full WASM rebuild on this crate will likely be in the 15–60 second range on a warm cache, longer on cold. dx serve's element & asset hot-reload covers RSX-only edits without a recompile, and Subsecond hot-patching (--hotpatch) cuts patchable changes to seconds — but the latter is experimental in 0.7, and most refactors trigger a full rebuild anyway. This is the single biggest DX regression of the migration; the team will feel it from day one.

Mitigations:

  • Run cargo check -p pastedev-web --target wasm32-unknown-unknown in a tight loop (e.g. bacon) for fast feedback on compile errors without re-bundling.
  • Use dx serve's default element/asset hot-reload for class-string + markup edits — those don't recompile.
  • Try --hotpatch for tight iteration on a single component; fall back to full dx serve when it gets unstable.
  • Split crates/web internals into smaller modules so incremental compilation has more to skip. Avoid one giant app.rs.
  • Crank up opt-level for dev only on the heaviest dependencies in [profile.dev.package.*] (e.g. pulldown-cmark) if rebuild dwarfs link time.
  • Set the team's expectation in the PR description: dev rebuilds are seconds, not milliseconds. Don't compare to Vite as a parity goal — it isn't one we can win.

Non-risks we're not worried about

  • Browser support. WASM-MVP support is now universal in our target browsers (Chromium, Firefox, Safari ≥ 14). The Vue SPA already requires modules & ES2020.
  • SEO of slug routes. Already addressed server-side — the per-snippet meta tags are injected into the shell HTML before WASM ever runs. Bots see them; the WASM mount doesn't disturb them.
  • Auth/session model. No changes — same cookie, same extractors. Whatever works today keeps working.
  • Bundle hash invalidation. dx emits hashed filenames; rust-embed ships them; cache-busting is automatic.

What we'd revisit post-cutover

  1. Add a Wasm-Split boundary on the admin queue; admins are < 1% of sessions.
  2. Consider porting the highlight worker to Rust if a future syntect tree-sitter approach fits in < 200 kB compressed.
  3. Look at Dioxus Native (WGPU) for a desktop app pinned to the same component tree.
